# Research Skill — Standard Operating Procedure

*Version 1.0 — 2026-02-05*
*Sources: meta-research-methodology.md, synthesis-and-research-roles.md, gold-standards-library.md*

---

## Warum dieser Skill existiert

Jede Research-Aufgabe ohne Prozess produziert mittelmäßige Ergebnisse, verschwendet Tokens und erfordert Nacharbeit. Dieser Skill definiert den Standard: wie wir researchen, synthetisieren, dokumentieren und compunden.

---

## Phase 0: GOLD STANDARD CHECK (VOR allem anderen)

**Nie von Null anfangen.**

1. **Gibt es eine Norm?** ISO, DIN, PRISMA, APA, EQUATOR → Orientierung daran
2. **Wer macht das am besten?** 3-5 Best-in-Class Beispiele identifizieren
3. **Was haben die gemeinsam?** Struktur, Länge, Ton, Format extrahieren
4. **Template ableiten** → Das ist der Startpunkt, nicht die leere Seite

**Referenz:** `research/night-research/gold-standards-library.md` — 15 Output-Typen mit Normen + Beispielen

**Regel:** Wenn kein Gold Standard bekannt → erst 10 Min recherchieren bevor der erste Buchstabe geschrieben wird.

## Phase 1: Research Design (BEVOR recherchiert wird)

### 1.1 Frage definieren
- Was genau wollen wir wissen?
- Welche Entscheidung hängt davon ab?
- Wer ist der Empfänger? Was erwartet er?

### 1.2 Scope definieren
- Breite vs. Tiefe: Was ist wichtiger?
- Zeitrahmen: Wie aktuell müssen Quellen sein?
- Sprache: DE, EN, beides?
- Max. Zeitbudget für diese Research

### 1.3 Hypothese formulieren (McKinsey-Methode)
- "Ich glaube die Antwort ist X weil Y"
- Research bestätigt oder widerlegt die Hypothese
- Verhindert zielloses Sammeln

### 1.4 Stopping Criteria definieren
- Wann ist genug? (z.B. "3 unabhängige Quellen bestätigen dasselbe")
- Data Saturation: Wenn neue Quellen nichts Neues bringen → aufhören
- Zeitlimit einhalten — Perfekte Research die zu spät kommt ist wertlos

## Phase 2: Quellen sammeln

### 2.1 Quellenhierarchie (CIA Admiralty Code adaptiert)

| Tier | Quelle | Zuverlässigkeit |
|------|--------|----------------|
| A | Primärquellen (Studien, offizielle Reports, Daten) | Höchste |
| B | Experten-Analysen (McKinsey, RAND, Contrary Research) | Hoch |
| C | Qualitätsjournalismus (FT, Bloomberg, TechCrunch) | Mittel-Hoch |
| D | Blogs, Podcasts, Social Media von Experten | Mittel |
| E | Aggregatoren, Wikipedia, AI-generierter Content | Niedrig |
| F | Anonyme Quellen, Gerüchte, unverifizierbares | Nicht verwenden |

### 2.2 Triangulation
- **Minimum 3 unabhängige Quellen** für jede Kernaussage
- Wenn nur 1-2 Quellen → explizit als "niedrige Konfidenz" markieren
- Widersprüche aktiv suchen, nicht ignorieren

### 2.3 Such-Strategie
- Bekannte Quellen zuerst (Gold Standard Referenzen)
- Dann breit suchen (Web Search)
- Dann tief gehen (Fetch einzelner Artikel)
- Spezial-Quellen je nach Thema (GoVCLab, Decile Base, PitchBook, Arxiv)

## Phase 3: Synthesis (DAS Herzstück)

### 3.1 Synthesis ≠ Summary

| Summary | Synthesis |
|---------|-----------|
| Komprimiert eine Quelle | Verbindet mehrere Quellen |
| "Quelle A sagt X" | "X + Y zusammen bedeuten Z" |
| Berichtet Fakten | Erzeugt neue Erkenntnis |
| Jeder kann es | Erfordert Denken |

### 3.2 McKinsey "So What?" Test (3 Ebenen)

Jede Erkenntnis muss 3 Fragen bestehen:
1. **So What?** — Warum ist das wichtig?
2. **Now What?** — Was folgt daraus als Aktion?
3. **Says Who?** — Welche Evidenz stützt das?

Wenn eine Erkenntnis den "So What?" Test nicht besteht → raus.

### 3.3 Insight Ladder

```
Daten → Observation → Finding → Insight → Implication → Recommendation
"30% nutzen AI" → "Adoption steigt" → "Vor allem in Enterprise" → 
"Enterprise kauft AI als Effizienz-Tool" → "Vertikale AI-Lösungen gewinnen" → 
"Investiere in vertikale AI für Enterprise"
```

**Ziel:** Mindestens auf "Insight" Ebene ankommen. Observations und Findings sind Zwischenprodukte, keine Endergebnisse.

### 3.4 Widersprüche = Gold

Wenn Quellen sich widersprechen → NICHT eine ignorieren.
- Dokumentiere beide Positionen
- Analysiere WARUM sie sich widersprechen (Methodik? Zeitraum? Definition?)
- Der Widerspruch selbst ist oft die wertvollste Erkenntnis

### 3.5 Synthesis-Rollen für Sub-Agents

| Rolle | Aufgabe | Wann einsetzen |
|-------|---------|---------------|
| Researcher | Quellen sammeln, Fakten extrahieren | Immer |
| Synthesizer | Querverbindungen finden, Insights generieren | Bei 3+ Quellen |
| Critic / Red Team | Schwächen finden, Gegenargumente liefern | Bei wichtigen Entscheidungen |
| Devil's Advocate | "Was wenn das Gegenteil stimmt?" | Bei hohem Risiko |

**Anthropic-Research zeigt:** Multi-Agent mit Rollen = 90% besser als Single-Agent.

## Phase 4: Output erstellen

### 4.1 Format nach Empfänger

- **Florian direkt:** Kurz, meinungsstark, Recommendation zuerst
- **Extern (Investor, Kunde):** Gold Standard Template verwenden
- **Intern (Skill, Standard):** Strukturiert, mit Checklisten
- **Sub-Agent:** Ultra-fokussierte Task-Beschreibung

### 4.2 Pyramid Principle (Minto)

```
1. Recommendation/Answer ZUERST
2. Supporting Arguments (3 max)
3. Evidence für jedes Argument
```

Nie umgekehrt. Der Leser will die Antwort, nicht die Reise.

### 4.3 Qualitäts-Checkliste vor Delivery

- [ ] Beantwortet die Research die ursprüngliche Frage?
- [ ] Besteht jede Kernaussage den "So What?" Test?
- [ ] Sind mindestens 3 Quellen trianguliert?
- [ ] Sind Widersprüche dokumentiert?
- [ ] Ist das Format Gold-Standard-konform?
- [ ] Ist die Länge angemessen? (kürzer ist IMMER besser)
- [ ] Würde Florian das OHNE ÄNDERUNGEN nutzen?

## Phase 5: Dokumentieren & Vernetzen

### 5.1 Skill/Standard updaten
- Was hat bei dieser Research funktioniert? → Diesen Skill updaten
- Was war ein neuer Trick? → `agents/CONTINUOUS-IMPROVEMENT.md`
- Was war ein Fehler? → Hier unter "Known Pitfalls" ergänzen

### 5.2 Obsidian-Vernetzung
- **Wo gehört das Wissen hin?** → Relevanten Obsidian-Ordner identifizieren
  - `20-Knowledge/` für Evergreen-Wissen
  - `10-Projects/` für projektspezifisches
  - `60-Lessons/` für Learnings
- **Was verbindet sich?** → Querverweise notieren
  - "Diese CNC-Research verbindet sich mit Ainary-Ventures Fund Thesis"
  - "Diese LP-Strategie ergänzt den VC Lab Sprint 2"
- **Obsidian-Note erstellen oder updaten** wenn das Wissen langfristig wertvoll ist

### 5.3 Memory updaten
- Signifikante Erkenntnisse → `memory/YYYY-MM-DD.md`
- Strategische Insights → `MEMORY.md`
- Offene Fragen → Als Task dokumentieren

### 5.4 Messen
- Wie lange hat die Research gedauert? (Gantt-Eintrag)
- Wie viele Iterationen bis zur Annahme? (Ziel: ≤2)
- Hat Florian es ohne Änderungen genutzt? (Ziel: Ja)

## Rollen-Konfiguration für Sub-Agents

### Einfache Research (1 Thema, <30 Min)
→ Selbst machen. Kein Agent nötig.

### Mittlere Research (2-5 Themen, 30-60 Min)
→ 1-2 Researcher-Agents parallel, Synthesis selbst.

### Tiefe Research (5+ Themen, >60 Min)
→ Researcher-Agents parallel + Synthesizer-Agent + eigene Review
→ Aufgabe pro Agent: 1 klar definiertes Thema, explizites Output-Format

### Agent-Task Template
```
Recherchiere [THEMA].

Suche nach:
1. [Spezifische Frage 1]
2. [Spezifische Frage 2]
3. [Spezifische Frage 3]

Für jede Frage: mindestens 3 Quellen, URLs angeben.

Output-Format: [Beschreibung]
Schreibe nach: [Dateipfad]
```

## Known Pitfalls

| Fehler | Lösung |
|--------|--------|
| Von Null anfangen | Phase 0: Gold Standard Check |
| Zu breit recherchieren | Hypothese + Stopping Criteria vorher |
| Summary statt Synthesis | "So What?" Test auf alles |
| Nur bestätigende Quellen | Aktiv nach Widersprüchen suchen |
| Zu lange Reports | Pyramid Principle: Antwort zuerst |
| Research ohne Dokumentation | Phase 5 ist PFLICHT, nicht optional |
| Sub-Agents ohne klare Aufgabe | Agent-Task Template verwenden |
| Perfekte Research, zu spät | Zeitbudget definieren und einhalten |

## Appendix: Infographic Typography Standards

**Source:** WCAG 2.1, Wix Design Guide, Substack Official Specs, Pentagram/Deloitte Insights system

### Font Size Minimums (Screen/Digital Infographics)

| Element | Minimum | Recommended | Notes |
|---------|---------|-------------|-------|
| Headline/Title | 28px | 32-48px | Font-weight 800-900, letter-spacing tight |
| Overline/Label | 10px | 11-13px | ONLY with font-weight 700 + letter-spacing 2-3px + uppercase |
| Body/Description | 14px | 16-18px | 14px absolute minimum for body text |
| Metrics/KPIs | 12px | 14-16px | Bold (600-700) helps at smaller sizes |
| Captions/Source | 10px | 11-12px | Only for non-essential info (brand, source line) |
| Stats/Numbers (hero) | 36px | 40-64px | Font-weight 900 for impact |

**Rule:** Nothing below 10px. Body text never below 14px. If in doubt, go larger.

### Substack Inline Image Specs

| Element | Dimensions |
|---------|-----------|
| Inline content image | **1100px wide** (Substack content width) |
| Aspect ratio | No fixed requirement, but 1100×700 to 1100×800 works well |
| Thumbnail (OG) | 1200×630px (social sharing standard) |
| Resolution | **2x (retina)** — render at 2200px wide, display at 1100px |
| File format | PNG for graphics with text, JPG for photos |

**Rule:** Always render at 2× with `--force-device-scale-factor=2`. Substack displays at 1100px max width — design at 1100px, render at 2200px actual pixels.

### Typography Scale (for 1100px-wide infographics)

```
Title:      32px / weight 900 / -1px tracking
Subtitle:   15-16px / weight 400 / color #94A3B8
Overline:   11px / weight 700 / 3px tracking / uppercase
Section:    10px / weight 700 / 2.5px tracking / uppercase
Body:       14-15px / weight 400-500
Metric:     14px / weight 600-700 / color accent
Stat Hero:  40-64px / weight 900 / -1px tracking
Source:     10-11px / weight 500 / color #CBD5E1
Brand:      10px / weight 600 / 1px tracking
```

### Graphic Design Elements (Editorial Infographics)

**Source:** McKinsey/Wolff Olins 2019 rebrand, Pentagram/Deloitte Insights system, Giorgia Lupi

**Problem:** Text-only infographics look like tables, not graphics. Subtle visual elements elevate from "information" to "designed information."

**Professional graphic elements to use:**

| Element | Purpose | CSS Implementation |
|---------|---------|-------------------|
| Accent bar (3-4px) | Marks sections, creates hierarchy | `border-left: 4px solid #2563EB` |
| Progress dots | Shows sequence, numbered steps | Colored circles with numbers |
| Connector lines | Shows flow, relationship | `border-bottom: 2px solid` between elements |
| Vertical accent | Marks importance | Left border on key insight boxes |
| Circle/dot marker | "Look here" signal (Deloitte green dot) | Small colored circle next to key data |
| Subtle gradient | Adds depth (McKinsey dark slides) | `background: linear-gradient(...)` |
| Icon-sized shapes | Replaces emoji, adds visual rhythm | CSS shapes (circles, squares, triangles) |
| Grid lines | Background structure (Pentagram/Deloitte) | Very subtle 1px lines at 5% opacity |
| Brand color accent | Sparing use, max 10% of surface | Only on key data, CTAs, hero numbers |
| Alternating backgrounds | Section separation | White / light gray (#F8FAFC) alternation |

**Rules:**
1. **Max 2 graphic elements per section** — restraint is the skill
2. **Every graphic element must serve a purpose** — decoration = amateur
3. **Accent color on max 10% of surface** — McKinsey uses vivid blue sparsely
4. **Use shapes to create visual rhythm** — dots, bars, lines break up text
5. **No emojis, ever** — use CSS shapes or nothing

### Recurring Contrast Errors (CROSS-REFERENCE)

**This error has appeared 3 times:**
1. 36ZERO Report: Yellow (#EAB308) on white = 1.9:1 ratio ❌
2. 36ZERO Report: Gray on gray backgrounds ❌
3. Substack infographics: #F1F5F9 step numbers on #FAFBFC = ~1.1:1 ratio ❌

**Rule:** NEVER use any color lighter than #94A3B8 (slate-400) on white/near-white backgrounds.

| Color | On white | Verdict |
|-------|----------|---------|
| #F1F5F9 | 1.1:1 | ❌ INVISIBLE |
| #CBD5E1 | 1.8:1 | ❌ TOO LIGHT |
| #94A3B8 | 3.0:1 | ⚠️ Only for non-essential |
| #64748B | 4.6:1 | ✅ Body text OK |
| #475569 | 7.1:1 | ✅ Good for all text |
| #0F172A | 17.5:1 | ✅ Maximum contrast |

**For decorative numbers (like step 01, 02):** Use min #CBD5E1 with font-weight 900 and large size (40px+), OR use #E2E8F0 background fill with #64748B text.
