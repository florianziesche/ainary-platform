03
Feature-Denken statt System-Denken
Ein Chatbot hier, eine Zusammenfassung dort. Drei Tools, drei Datenquellen, drei Wahrheiten. Das skaliert nicht.
WAS FUNKTIONIERT
Eine Ontologie. Eine Datenquelle. Ein System.
Feature-Denken vs. System-Denken
Feature-Denken
Pro Feature eine eigene Datenquelle
Keine Evidenz-Markierung auf Claims
Lücken werden verschwiegen
Kein Kill Switch vorhanden
Audit ist ein Nachgedanke
Skalierung: O(n²) pro Feature
Jeder onclick ist handgeschrieben
"Funktioniert bei mir"
vs.
System-Denken
Eine Ontologie für alles
E/I/J/A auf jedem Claim
Lücken explizit offengelegt
Kill Switch eingebaut
Audit in der Render-Pipeline
Skalierung: O(1) pro Feature
Wenn es rendert, ist es ein Object
"Hier ist der Beweis"
Palantir Ontology-Prinzip: Schema vor Code. Daten ≠ Präsentation.
Das Geheimnis: Ehrlichkeit als System.
Jeder Claim wird klassifiziert. Nicht "wahr/falsch" — sondern wie sicher.
E
Empirisch
Belegt durch externe Quelle. Überprüfbar. Link vorhanden.
Beispiel: "570.000 Stellen unbesetzt" — dbb Monitor 2025
I
Inference
Abgeleitet aus Daten. Logisch, aber nicht direkt belegt.
Beispiel: "Windkraft 78% negativ" — 342 Posts analysiert
J
Judgment
Einschätzung basierend auf Erfahrung. Kann falsch sein.
Beispiel: "Quick Win möglich in 2-4 Wochen"
A
Assumption
Annahme. Keine Daten. Sollte hinterfragt werden.
Beispiel: "Nächste Antragsrunde Sommer 2026"
